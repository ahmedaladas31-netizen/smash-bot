# مراجعة إنتاجية — بوت واتساب سماش لاب

> المراجعة مبنية على فحص الـ JSON الفعلي للـ workflows الثلاثة في n8n
> (`My workflow 2`، `Smash Lab - Send Messages`، `Smash Lab - Error Handler`)
> وكود الداشبورد — مش على وصف التقرير فقط. التاريخ: 2026-06-12.

## ملخص تنفيذي

النظام مبني بشكل جيد والتدفق الأساسي سليم، لكن فيه **5 مشاكل حرجة لازم تنحل قبل
الإطلاق الفعلي**، أهمها: webhook الإرسال مفتوح بدون مصادقة، حلقة أخطاء لا نهائية
محتملة في الـ Error Handler، وتسريب ماركرات مكسورة للزبائن (وهذا صار فعلاً —
الدليل إن كود Parse Order يعالج `[BORDER]` و`[ORDR]`). الـ system prompt
المحسّن جاهز في `system-prompt-v2.txt` وسكريبت الاختبار في `test-script.mjs`.

## 1. مشاكل حرجة (قبل الإطلاق)

### 1.1 — webhook الإرسال `smash-send` مفتوح بدون أي مصادقة 🔴 أمني

الـ webhook في `Smash Lab - Send Messages` يستقبل POST بدون أي تحقق:
أي شخص يعرف الرابط `https://darkbyte.app.n8n.cloud/webhook/smash-send` يقدر
**يرسل أي رسالة واتساب لأي رقم باسم المطعم** (احتيال، انتحال، إزعاج زبائن).

**الحل:**
1. أضف شرط في n8n بعد الـ webhook: IF
   `{{ $json.headers['x-smash-token'] === 'TOKEN_سري_طويل' }}` → أكمل، وإلا توقف.
2. في الداشبورد عدّل [webhook.ts](../src/lib/webhook.ts) ليرسل الهيدر:
   ```ts
   headers: {
     'Content-Type': 'application/json',
     'X-Smash-Token': import.meta.env.VITE_WEBHOOK_TOKEN,
   }
   ```
   ملاحظة: التوكن سيظهر في bundle المتصفح (الداشبورد client-side)، لكن مع كون
   الداشبورد داخلي للمطعم هذا مقبول كطبقة أولى. الحل الأمتن لاحقاً: تمرير
   الإرسال عبر Supabase Edge Function.

### 1.2 — حلقة لا نهائية محتملة في الـ Error Handler 🔴

`Smash Lab - Error Handler` هو الـ error workflow لكل الـ workflows بما فيها
`Send Messages` نفسه، وهو يبلّغ عن الخطأ عبر **POST إلى `smash-send`**.
السيناريو: واتساب API تعطّل مؤقتاً → `Send Messages` يفشل → الـ Error Handler
يرسل تنبيه عبر `smash-send` → يفشل هو كمان → error جديد → تنبيه جديد → **حلقة
بلا توقف** (الـ webhook يرد 200 فوراً بسبب `responseMode: onReceived` فالـ HTTP
ينجح والفشل يصير جوّا).

**الحل (اختر واحد):**
- الأفضل: استبدل عقدة HTTP Request في الـ Error Handler بعقدة WhatsApp مباشرة
  (نفس عقد الإرسال في البوت الرئيسي) — يكسر الاعتماد على `smash-send`.
- أو: أضف IF قبل الإرسال: `{{ $json.workflow.id !== 'eFUIDSM893VfYRaZ' }}`
  (تجاهل أخطاء Send Messages نفسه).

### 1.3 — تسريب الماركرات المكسورة للزبون 🔴 تجربة زبون

كود `Parse Order` يعالج 4 صيغ: `[ORDER]`, `[BORDER]`, `[ORDR]`, `[ORDER ]` —
معناها GPT-4o أخطأ في كتابة الماركر سابقاً. المشكلة: عقدة `Order Confirmed?`
تفحص `[ORDER]` و`[BORDER]` فقط. لو الموديل كتب `[ORDR]` أو `[ORDER ]` أو أي
صيغة مكسورة أخرى، الرسالة تسقط لفرع `Send message` الذي يرسل **الناتج الخام
كاملاً** — فيوصل للزبون نص مثل: `[ORDR]{"items":"...","total":65,...}`.

**الحل (طبقتين):**
1. البرومبت v2 يشدد على الصيغة الحرفية للماركرات (قسم "الماركرات" الجديد).
2. شبكة أمان في الـ workflow: عدّل `textBody` في عقدة `Send message` إلى:
   ```
   ={{ ($json.output || '')
        .replace(/\[[A-Z][A-Z_ ]*\]\s*\{[\s\S]*$/g, '')
        .replace(/\[[A-Z][A-Z_ ]*\]/g, '')
        .trim() || 'وصلتنا رسالتك! كيف بنقدر نساعدك؟' }}
   ```
   هكذا أي ماركر (سليم أو مكسور) وأي JSON بعده يُحذف قبل الإرسال للزبون.

### 1.4 — صمت تام إذا ما في طلب في مساري ASK_STATUS و ASK_TIME 🔴

`Get Latest Order For Status` و`Get Order For Time` لو رجّعوا 0 صفوف (زبون
جديد يسأل "وين طلبي" بدون ما يكون طلب)، العقد التالية ما بتشتغل والزبون
**ما يستلم أي رد إطلاقاً** — أسوأ تجربة ممكنة في واتساب.

**الحل:** فعّل "Always Output Data" على عقدتي Supabase، وأضف IF بعد كل وحدة:
`{{ $json.id }}` exists → أكمل المسار الحالي، وإلا → عقدة WhatsApp ترد:
"ما لقيت عندك طلب فعّال هلأ. بتحب تطلب شي؟"

### 1.5 — الإلغاء يلغي طلب جاهز/بالطريق، ويكذب لو ما في طلب 🔴 منطق عمل

`Cancel Order In DB` فلتره `status=neq.delivered&status=neq.cancelled` فقط —
الزبون يقدر يلغي طلب **جاهز** أو **طالع للتوصيل** بعد ما المطبخ خلصه وخسّر
المطعم. ولو ما عنده طلب أصلاً، الـ update يصيب 0 صفوف لكن الزبون يستلم رسالة
"تم إلغاء طلبك" — معلومة غلط.

**الحل:**
1. غيّر الفلتر إلى: `status=in.(new,preparing)`.
2. فعّل Always Output Data وافحص الناتج: لو رجع صف → رسالة الإلغاء الحالية.
   لو ما رجع شي → "طلبك إما خلص تحضيره أو ما في طلب فعّال — احكينا عالمطعم
   مباشرة وبنظبطها."

## 2. مشاكل عالية الأولوية

### 2.1 — زر "وضع الضغط" في الداشبورد لا يؤثر على البوت إطلاقاً 🟠

الداشبورد يكتب `busy_mode` و`busy_delay` ([api.ts:73-90](../src/lib/api.ts#L73-L90))
ويعرض "بنستقبل الطلبات بعد X دقيقة"، لكن البوت يقرأ **`general_wait_time` فقط**.
الموظف يفعّل وضع الضغط ويظن أن الزبائن انبلغوا — والبوت يكمل عادي بوقت 20 دقيقة.

**الحل:** في عقدة AI Agent غيّر سطر الوقت في الـ text إلى وقت فعلي موحّد:
```
وقت التحضير العام الحالي للمطعم: {{ $json.busy_mode
  ? Math.max($json.busy_delay || 30, $json.general_wait_time || 20)
  : ($json.general_wait_time || 20) }} دقيقة.
```
هكذا تفعيل وضع الضغط يرفع الوقت تلقائياً، ولو وصل 40+ تشتغل آلية
"بتحب تستنى؟" الموجودة.

### 2.2 — نافذة الذاكرة 5 رسائل فقط (الافتراضي) 🟠

عقدة `Simple Memory` بدون `contextWindowLength` → الافتراضي **5 تفاعلات**.
محادثة طلب كاملة (صنف ← حجم ← وجبة/ساندويش ← توصيل ← اسم/عنوان) تتجاوز 5
بسهولة، فالموديل **ينسى بداية الطلب** قبل ما يحفظه — أخطر سبب صامت لطلبات
ناقصة أو خاطئة.

**الحل:** اضبط Context Window Length = 20 في عقدة Simple Memory.
(جرّب `node test-script.mjs --window 5` لتشوف المشكلة بنفسك مقابل `--window 20`.)

ملاحظة إضافية: الذاكرة لا تنتهي صلاحيتها — محادثة أمس تظل موجودة اليوم وقد
تشوّش (زبون يقول "تمام" اليوم والموديل يفسرها تأكيداً لطلب قديم). البرومبت v2
يعالج جزئياً ("سجّل الطلب مرة وحدة بس")، والحل الجذري مستقبلاً: session key
يتضمن التاريخ (مثلاً `{{ $json.messages[0].from + '_' + $now.format('yyyy-MM-dd') }}`)
مع قبول أن المحادثة "تنتسى" عند منتصف الليل.

### 2.3 — الرسائل الصوتية تُتجاهل بصمت 🟠

`Filter Messages` يمرر النصوص فقط ويرمي الباقي. في فلسطين نسبة كبيرة من
الزبائن ترسل **رسائل صوتية** — هؤلاء لن يستلموا أي رد أبداً.

**الحل (مرحلة أولى):** فرع FALSE من الفلتر → IF نوع الرسالة
`{{ ['audio','image','video','document'].includes($json.messages[0].type) }}`
→ عقدة WhatsApp ترد: "أهلين! هلأ بستقبل رسائل كتابية بس — اكتبلي طلبك
وبخدمك فوراً." (تجاهل reactions/stickers حتى لا نزعج).
مستقبلاً: تحويل الصوت لنص عبر Whisper ثم نفس المسار.

### 2.4 — تكرار حفظ الطلب (duplicate orders) 🟠

ما في أي حماية من التكرار: لو الزبون كتب "تمام" مرتين أو أعاد التأكيد، الموديل
قد يصدر `[ORDER]` ثانية ← صفّان في `orders` ← المطبخ يحضّر الطلب مرتين.

**الحل:**
1. البرومبت v2 يتضمن: "سجّل الطلب مرة وحدة بس".
2. شبكة أمان: قبل `Save Order` أضف عقدة Supabase تجيب آخر طلب لنفس الرقم خلال
   آخر 3 دقائق بحالة `new`؛ لو موجود → تخطَّ الإدخال وأرسل "طلبك محفوظ من قبل،
   عم نشتغل عليه".

### 2.5 — الداشبورد لا يعرف أن رسالة الواتساب فشلت 🟠

`Receive Send Request` مضبوط `responseMode: onReceived` → يرد 200 فوراً حتى لو
فشل الإرسال الفعلي بعدها. الداشبورد يعتمد على فشل HTTP ليسجل
`message_sent=false` ([App.tsx](../src/App.tsx)) — وهذا لن يحدث أبداً.

**الحل:** غيّر Response Mode إلى "When Last Node Finishes" — عندها فشل عقدة
الواتساب يرجع خطأ HTTP للداشبورد ويشتغل مسار `markMessageNotSent` الموجود.

### 2.6 — قيود نافذة الـ 24 ساعة في WhatsApp Business 🟠

رسائل الداشبورد (جاهز/بالطريق/ملغي) هي رسائل يبدأها العمل — مسموحة فقط خلال
**24 ساعة من آخر رسالة من الزبون**. ضمن تدفق مطعم سريع هذا نادر الحدوث، لكن
طلب مؤجل أو رسالة متأخرة سترفض بخطأ `131047` بصمت.

**الحل:** بعد إصلاح 2.5، الفشل سيظهر للداشبورد ويتسجل `message_sent=false`.
للمدى الطويل: template message معتمدة من Meta للتحديثات المتأخرة.

## 3. مشاكل متوسطة

### 3.1 — "وين طلبي" يجاوب عن طلب قديم جداً

`Get Latest Order For Status` بدون فلتر زمني ولا فلتر حالة: زبون آخر طلب له
قبل أسبوع بحالة `ready` (نسي الموظف يحدثه) يسأل "وين طلبي" → "طلبك جاهز وبتقدر
تستلمو"! **الحل:** أضف للفلتر `&created_at=gte.` + تعبير لآخر 24 ساعة:
```
={{ 'customer_phone=eq.' + $('WhatsApp Trigger').item.json.messages[0].from
   + '&created_at=gte.' + new Date(Date.now() - 24*60*60*1000).toISOString()
   + '&order=created_at.desc' }}
```
مع معالجة "ما في نتيجة" من بند 1.4. كذلك أضف فرع `delivered` في
`Status Switch` برد: "طلبك وصل، صحتين وعافية!" (حالياً يسقط في الـ fallback).

### 3.2 — الطلب المتأخر يقال له "ثانية بيكون عندك"

في `Send Prep Time Reply`: لو `remaining <= 2` (بما فيها السالب — طلب متأخر
نص ساعة) الرد "طلبك قريب جداً، ثانية بيكون عندك". **الحل:** أضف حالة ثالثة:
`remaining < -5` → "طلبك بمراحله الأخيرة، دقايق ويكون معك — سامحنا عالتأخير"
(+ يفضّل تحديث `customer_asked=true` ليظهر للموظف).

### 3.3 — `total` قد يصل NaN لقاعدة البيانات

في `Parse Order`: لو الموديل كتب `"total":"65 شيكل"` → `Number(...)` = NaN →
فشل إدخال في Supabase (error workflow يشتغل والزبون بلا تأكيد). **الحل** — في
كود Parse Order استبدل سطر total بـ:
```js
let t = data.total;
if (typeof t === 'string') t = parseFloat(t.replace(/[^\d.]/g, ''));
total = Number.isFinite(t) ? t : null;
```

### 3.4 — تحديث Supabase بـ `limit=1` على PATCH

`Mark Customer Asked` / `Mark Time Asked` / `Cancel Order In DB` تستخدم
`order=created_at.desc&limit=1` في **update**. PostgREST يدعم limited update
في الإصدارات الحديثة فقط وبشروط؛ لو تجاهل limit سيحدّث **كل** الطلبات الفعالة
لنفس الرقم. **إجراء:** اختبرها مرة بزبون عنده طلبين فعالين وتأكد أن صفاً واحداً
فقط تغيّر. البديل الآمن: get ثم update بـ `id=eq.{id}`.

### 3.5 — سباق التنفيذ المتوازي

رسالتان سريعتان من نفس الزبون = تنفيذان متوازيان (ذاكرة متسابقة، احتمال طلبين).
نادر لكن وارد. حل n8n بسيط غير متاح مباشرة — اكتفِ بحماية التكرار (2.4)
وراقب. (الحل الجذري: صف انتظار لكل رقم — Phase C.)

### 3.6 — نص الموديل يُرمى في فرع المنيو

عند `[SEND_MENU]` تُرسل الصور فقط وأي نص كتبه الموديل قبل الماركر يُحذف.
سلوك مقصود والبرومبت يقول "اكتب فقط [SEND_MENU]" — فقط للعلم، السكريبت يختبره
(سيناريو 4).

- طرق الدفع (كاش فقط؟) — نفس المعاملة حالياً.

## 5. تحسينات الـ routing والـ error handling (خارطة)

ترتيب التنفيذ المقترح (الأرقام تشير للبنود أعلاه):

**اليوم (قبل أي إطلاق):**
1. مصادقة `smash-send` (1.1) + تعديل webhook.ts بالداشبورد
2. كسر حلقة الـ Error Handler (1.2)
3. sanitizer الماركرات في `Send message` (1.3)
4. حراسة النتائج الفارغة في مساري الحالة والوقت (1.4)
5. تقييد الإلغاء بـ `new,preparing` + رد صادق (1.5)
6. لصق البرومبت v2 وتشغيل `test-script.mjs` للتحقق

**هذا الأسبوع:**
7. ربط busy_mode بالبوت (2.1)
8. Context Window = 20 (2.2)
9. رد على الرسائل الصوتية (2.3)
10. responseMode = lastNode في Send Messages (2.5)
11. حماية التكرار قبل Save Order (2.4)

**تحسينات لاحقة:** بنود القسم 3 + Phase C أدناه.

### Phase C (اقتراح معماري)

استبدل نظام الماركرات النصية كلياً بـ **OpenAI function calling**: عرّف أدوات
`save_order(items, total, delivery_type, address, name)`، `cancel_order()`،
`get_order_status()`، `send_menu()` كـ sub-workflows في n8n. المزايا: JSON
مضمون البنية (schema-validated)، لا parsing ولا ماركرات مكسورة، ولا تسريب
للزبون. هذا يلغي بنود 1.3 و3.3 من جذورها. كلفته إعادة هيكلة الـ workflow،
لذلك هو مرحلة قادمة وليس شرط إطلاق.

## 6. الأمان — جواب سؤال SQL Injection

**الخلاصة: الخطر منخفض جداً، لا يحتاج تغييراً.**
- فلاتر Supabase المبنية بـ string concatenation تمر عبر **PostgREST query
  string** وليس SQL خام — أسوأ حالة هي تلاعب بالفلترة، ليس حقن SQL.
- المصدر الوحيد للقيمة هو `messages[0].from` من WhatsApp Trigger — رقم هاتف
  تتحقق منه Meta، وليس نص حر من الزبون. لا يوجد أي مكان يُدخل فيه نص رسالة
  الزبون في فلتر.
- **القاعدة للمستقبل:** لا تُدخل أبداً نص رسالة الزبون أو ناتج الـ AI في
  filter string. الوضع الحالي ملتزم بها.

البند الأمني الفعلي الوحيد هو 1.1 (webhook بلا مصادقة)، ويُذكر أيضاً: تأكد أن
RLS مفعّل على جداول Supabase بما يناسب الـ anon key المستخدم في الداشبورد.
