## المشكلة 1 — OpenAI Rate limit + فقدان السياق مع الرسائل المتتالية

**الجذر:** كل رسالة من الزبون = تنفيذ منفصل + استدعاء LLM منفصل بالتوازي. الزبون
اللي بيبعت 6 فقاعات ورا بعض = 6 استدعاءات بنفس اللحظة = ضغط rate limit + البوت
بيرد على الفقاعة الأولى والباقي لسا جاي (فقدان سياق). وإعدادات الـ retry كانت
محطوطة غلط جوّا `parameters` فكانت **معطّلة**.

**الحل المطبَّق:**
1. **Debounce** — عقدتين جداد على المسار الرئيسي:
   - `Debounce Wait` (Wait 8 ثواني) بعد `Filter Messages`.
   - `Debounce Gate` (Code) — بيستعلم جدول `messages`:
     - لو إجت رسالة **أحدث** من رسالتي خلال الانتظار → `return []` (يوقف بصمت، الأحدث رح يتكفّل بالكل).
     - وإلا → بيجمع كل الرسائل الواردة **من بعد آخر رد للبوت** بنص واحد (`combinedText`).
     - fail-safe: أي خطأ بالاستعلام → بيكمل برسالة الزبون الحالية لحالها (ما بيوقف الرد أبداً).
   - المسار: `Filter Messages → Debounce Wait → Debounce Gate → Get Restaurant Settings → AI Agent`.
   - `Log Inbound` ظل متفرّع مباشرة من `Filter Messages` (بيتسجّل فوراً قبل الانتظار، عشان الـ Gate يقراه).
   - `AI Agent.text` السطر الأخير صار يقرأ `$('Debounce Gate').item.json.combinedText` بدل رسالة وحدة.
2. **retry فعلي على عميل OpenAI** — `OpenAI Chat Model.options`: `maxRetries=6` + `timeout=60000`.
   عميل OpenAI بيعمل exponential backoff على 429 (بيحترم `Retry-After`) — أقوى من retry الـ n8n.

**الأثر:** 6 فقاعات بترجع استدعاء LLM واحد → ضغط أقل + البوت بيقرأ القصة كاملة قبل ما يرد.

## المشكلة 2 — طلب ناقص/صفر شيكل + تأكيد كاذب

**الجذر:** البوت بيطلّع `[ORDER]` قبل ما يكمل المعلومات، والـ workflow بيحفظ أي إشي
وبيقول للزبون "استلمنا طلبك" حتى لو `total=0` أو ناقص اسم/نوع/عنوان.

**الحل المطبَّق (بوابة تحقق حتمية):**
- `Parse Order` صار يحسب `isOrderValid` + `missingPrompt`. غير صالح لو:
  `total<=0` أو items فاضي أو لا توصيل ولا استلام أو الاسم فاضي أو (توصيل بدون عنوان).
- عقدة `Order Valid?` (IF) بين `Is Duplicate Order?` (فرع لا-مكرر) و`Save Order`:
  - صالح → `Save Order` (المسار القديم).
  - غير صالح → `Send Order Incomplete` (بيسأل عن الناقص بلهجة، **بدون ما يحفظ وبدون ما يكذب**) → `Log Out Incomplete`.

**الأثر:** انتهى طلب الصفر شيكل والتأكيد الكاذب. الصنف المش موجود (total=0) بينمسك
وبيرجع للزبون سؤال استرجاع بدل "استلمنا طلبك".

## المشكلة 3 — سؤالين برسالة + فصحى + "سبرايت مش متوفر"

**الحل المطبَّق (system-prompt-v3.txt):**
- قسم جديد "الزبون بيبعت كذا رسالة ورا بعض" + "تخطّى اللي عطاك ياه" — يكمّل الـ debounce.
- قاعدة سؤال واحد مشدّدة بأمثلة الفشل الحقيقية: "وجبة ولا ساندويش؟ وكم جرام؟" و"كم جرام وتوصيل ولا استلام؟" ممنوعة صراحةً.
- المشروبات: ممنوع تقول لأي واحد منهن "مش متوفر".
- قسم جديد "متى تطلّع [ORDER]" = checklist إلزامي (أصناف+أحجام+نوع+توصيل+اسم+رقم+total>0) قبل أي ماركر، وممنوع "استلمنا طلبك" قبل اكتماله.
- صيغة الـ JSON بعد [ORDER] صارت مكتوبة حرفياً (يقلّل أخطاء الـ parsing).
- كلمات فصيحة إضافية ممنوعة + بدائل محكية.
