# الإصلاحات الحرجة المطبَّقة (القسم 1 من findings.md)

التاريخ: 2026-06-12. كل تعديلات n8n نُشرت (published) وهي فعّالة الآن.

## ⚠️ خطوة يدوية واحدة مطلوبة منك (دقيقة واحدة)

عمليات الـ MCP تضبط الـ parameters لكنها **لا تقدر تضبط إعداد العقدة
"Always Output Data"** (هو إعداد node settings مش parameter). هذا الإعداد ضروري
ليشتغل مسار "لا يوجد طلب" الذي أضفته في 1.4 و1.5. بدونه السلوك يرجع للحالة
القديمة (صمت عند عدم وجود طلب) — لكن بدون أي ضرر على المسار الطبيعي.

افتح **My workflow 2** في n8n، ولكل عقدة من الثلاث: افتحها ← تبويب
**Settings** ← فعّل **Always Output Data**:

1. `Get Latest Order For Status`
2. `Get Order For Time`
3. `Cancel Order In DB`

بعدها احفظ وانشر (Publish). خلص — صار كل شي شغّال 100%.

> لماذا؟ في n8n، أي عقدة تستقبل 0 عناصر لا تعمل. عقد Supabase ترجع 0 صفوف
> عند عدم وجود طلب، فلا يصل شيء لعقد التحقق الجديدة. تفعيل Always Output Data
> يجبر العقدة تُخرج عنصراً فارغاً `{}` ليصل لـ IF فيوجّهه لرسالة بديلة.

## ما طُبِّق تلقائياً

### 1.1 — تأمين webhook الإرسال 🔴 → ✅

- **الداشبورد** ([webhook.ts](../src/lib/webhook.ts)): يرسل الآن هيدر
  `X-Smash-Token` من `VITE_WEBHOOK_TOKEN`. أُضيف المتغيّر إلى
  [.env](../.env) و[.env.example](../.env.example).
- **n8n (Smash Lab - Send Messages):** أُضيفت عقدة IF باسم **Check Token**
  بين الاستقبال والإرسال — لا يُرسَل أي شي إلا إذا طابق الهيدر التوكن السري.
- التوكن الحالي: `d9368208aed86e3d49835f5a3e730cfee9b1e4d28e8c6b39`
  (نفس القيمة في .env وفي عقدة Check Token).

> بعد التعديل لازم **تعيد بناء/نشر الداشبورد** (`npm run build`) حتى يدخل
> التوكن في الـ bundle. لو غيّرت التوكن لاحقاً، غيّره في المكانين معاً
> (Check Token في n8n + VITE_WEBHOOK_TOKEN).

### 1.2 — كسر حلقة الـ Error Handler اللانهائية 🔴 → ✅

- أُضيفت عقدة IF **Skip Self Errors**: تتجاهل أخطاء workflow الإرسال نفسه
  (`workflow.id !== eFUIDSM893VfYRaZ`) فلا تتولّد حلقة تنبيهات لا تنتهي.
- أُضيف هيدر `X-Smash-Token` لعقدة **Send WhatsApp Alert** حتى تمرّ تنبيهات
  الأخطاء عبر حماية Check Token الجديدة.
- (أُصلِح بالمناسبة `specifyBody` من قيمة قديمة خاطئة `keypairs` إلى `keypair`.)

### 1.3 — منع تسريب الماركرات المكسورة للزبون 🔴 → ✅

- عقدة **Send message**: صار `textBody` ينظّف أي ماركر (سليم أو مكسور مثل
  `[ORDR]`/`[BORDER]`) وأي JSON يتبعه قبل الإرسال، ولو صار الناتج فاضي يرسل
  رسالة بديلة لطيفة بدل نص فارغ.

### 1.4 — حراسة النتائج الفارغة في مساري الحالة والوقت 🔴 → ✅ (+ خطوة يدوية)

- مسار "وين طلبي": أُضيفت IF **Has Active Order?** بعد
  `Get Latest Order For Status`. وجود طلب → Status Switch (كالعادة)؛
  لا طلب → عقدة جديدة **Send No Order Found**.
- مسار "قديش باقي": أُضيفت IF **Has Active Order For Time?** بعد
  `Get Order For Time`. وجود طلب → Has Prep Time?؛ لا طلب → Send No Order Found.
- **يحتاج تفعيل Always Output Data على العقدتين** (انظر الأعلى).

### 1.5 — تقييد الإلغاء + رد صادق 🔴 → ✅ (+ خطوة يدوية)

- **Cancel Order In DB**: الفلتر صار `status=in.(new,preparing)` — ما عاد
  ممكن إلغاء طلب جاهز أو طالع للتوصيل.
- أُضيفت IF **Cancel Matched?** بعده: نجح الإلغاء → Send Cancel Reply (كالعادة)؛
  لم يطابق أي صف → عقدة جديدة **Send Cancel Failed Reply** ("طلبك إما خلص
  وطلع للتوصيل أو ما في طلب فعّال…").
- **يحتاج تفعيل Always Output Data على Cancel Order In DB** (انظر الأعلى).

## التحقق بعد الخطوة اليدوية

أرسل من رقم اختبار للبوت:
1. "وين طلبي؟" بدون أي طلب سابق → لازم يجي رد "ما لقيت عندك طلب فعّال".
2. سجّل طلباً، خلّي الموظف يحوّله `ready` من الداشبورد، ثم "بدي ألغي" →
   لازم يجي رد فشل الإلغاء (مش "تم الإلغاء").
3. جرّب POST لـ smash-send بدون هيدر التوكن → لازم ما تنبعت أي رسالة واتساب.

# القسم الثاني — مشاكل عالية الأولوية (مطبَّق ومنشور 2026-06-12)

### 2.1 — ربط busy_mode بالبوت 🟠 → ✅
عقدة **AI Agent**: سطر "وقت التحضير العام" صار يحسب الوقت الفعلي:
`busy_mode ? max(busy_delay, general_wait_time) : general_wait_time`.
يعني تفعيل وضع الضغط من الداشبورد يرفع الوقت تلقائياً، ولو وصل 40+ تشتغل
آلية "بتحب تستنى؟" الموجودة أصلاً.

### 2.2 — نافذة الذاكرة 🟠 → ✅
عقدة **Simple Memory**: `Context Window Length = 20` (كانت 5 الافتراضية).
محادثة الطلب الكاملة ما عادت تتجاوز الذاكرة فينسى الموديل بدايتها.

### 2.3 — الرد على الرسائل غير النصية 🟠 → ✅
أُضيف فرع موازٍ من **WhatsApp Trigger**: عقدة IF **Is Media Message?**
(`type` ∈ audio/image/video/document) → عقدة **Send Text Only Notice** ترد:
"أهلين! هلأ بستقبل رسائل كتابية بس…". الرسائل النصية تمرّ عادي عبر المسار
القديم (Filter Messages)، والـ stickers/reactions تُتجاهَل كما هي.

### 2.4 — حماية تكرار الطلبات 🟠 → ✅ (بتصميم آمن)
أُضيفت عقدة Code **Dedup Check** بعد Parse Order: تستعلم Supabase REST عن طلب
بنفس الرقم بحالة `new` خلال آخر 3 دقائق. ثم IF **Is Duplicate Order?**:
مكرر → **Send Already Saved Reply** (تخطّي الحفظ)؛ غير مكرر → Save Order.

> **انحراف مقصود عن findings:** التقرير اقترح عقدة Supabase getAll. لكن عقدة
> getAll تُرجع 0 عناصر عند عدم وجود تكرار، فتُوقف السلسلة و**لن يُحفظ أي طلب
> غير مكرر إطلاقاً** (نفس قيد alwaysOutputData الذي لا تستطيع الـ MCP ضبطه).
> لذلك استخدمت عقدة Code بنمط **fail-open**: أي خطأ في التحقق (فشل الشبكة،
> منع fetch، RLS) → `isDuplicate=false` → يُحفظ الطلب عادي. النتيجة: الحماية
> تمنع التكرار عند نجاح الفحص، وما بتكسر الحفظ أبداً عند فشله.
> العقدة تستعمل مفتاح Supabase anon العام (نفسه بالداشبورد، للقراءة فقط).

### 2.5 — ظهور فشل الإرسال للداشبورد 🟠 → ✅
عقدة **Receive Send Request** (Send Messages): `Response Mode = lastNode`
(كان onReceived). الآن لو فشل إرسال الواتساب يرجع HTTP error للداشبورد،
فيشتغل مسار `markMessageNotSent` الموجود في [App.tsx](../src/App.tsx#L270-L276)
ويتسجّل `message_sent=false`. لا حاجة لأي تعديل في كود الداشبورد.

### 2.6 — نافذة الـ 24 ساعة 🟠 → ✅ (مغطّى بـ 2.5)
بعد 2.5 صار أي رفض بسبب نافذة الـ 24 ساعة (خطأ 131047) يظهر للداشبورد
ويتسجّل `message_sent=false`. الحل بعيد المدى (template معتمدة من Meta
للتحديثات المتأخرة) خارج نطاق هذا التعديل.

# القسم الثالث — مشاكل متوسطة (مطبَّق ومنشور 2026-06-12)

### 3.1 — "وين طلبي" يجاوب عن طلب قديم 🟡 → ✅
- عقدة **Get Latest Order For Status**: أُضيف فلتر `created_at=gte.` لآخر 24 ساعة،
  فما عاد يجاوب عن طلب من الأسبوع الماضي. (لو ما في طلب خلال 24 ساعة → مسار
  Send No Order Found من 1.4، يحتاج alwaysOutputData.)
- **Status Switch**: أُضيف فرع رابع `delivered` → عقدة جديدة **Send Delivered
  Reply** ("طلبك وصل، صحتين وعافية!"). أُعيد توصيل الـ fallback من المخرج 3
  إلى المخرج 4 بدقة (تحقّقت: out3→Delivered، out4→Mark Customer Asked + Send
  Status Reply).

### 3.2 — الطلب المتأخر يُقال له "ثانية بيكون عندك" 🟡 → ✅
- عقدة **Send Prep Time Reply**: أُضيفت حالة ثالثة — `remaining < -5` →
  "طلبك بمراحله الأخيرة، دقايق ويكون معك — سامحنا عالتأخير."
- أُضيفت عقدة **Mark Prep Asked** (Supabase، مقيّدة بـ id) كفرع موازٍ من
  Has Prep Time? (true)، تعلّم `customer_asked=true` فيظهر للموظف أن الزبون
  يسأل عن طلبه (كان مفقوداً على مسار الطلب المؤقّت بعكس مسار "لا وقت").

### 3.3 — total قد يصل NaN 🟡 → ✅
عقدة **Parse Order**: استُبدل سطر total بمنطق يفكّ النص الرقمي ويتحقق:
`typeof string → parseFloat(strip non-numeric)` ثم `Number.isFinite ? t : null`.
ما عاد ممكن كتابة NaN في عمود numeric (كان يفشل الإدخال ويشغّل error workflow).

### 3.4 — حدود `limit=1` على PATCH 🟡 → ✅ (جزئياً)
- **Mark Customer Asked** و**Mark Time Asked**: صار الفلتر `id=eq.{{ $json.id }}`
  (الطلب مجلوب مسبقاً في المسار، فالـ id متاح) بدل الفلتر بالهاتف+limit=1 —
  تحديث مقيّد بصف واحد مضمون، يزيل غموض limit-on-update.
- **Cancel Order In DB**: تُرك على فلتر الهاتف (لا يتوفّر id في مدخله — مدخله
  ناتج AI فقط) لكنه أصلاً `status=in.(new,preparing)&limit=1`. الخطر الوحيد
  المتبقّي: زبون عنده طلبان فعّالان معاً (نادر). للتحسين الكامل لاحقاً:
  get-then-update، لكنه يعيد مشكلة النتيجة الفارغة.

### 3.5 — سباق التنفيذ المتوازي 🟡 → موثّق (لا تغيير)
لا حل بسيط في n8n. مخفّف عملياً بحماية التكرار (2.4، fail-open). يبقى تحت
المراقبة؛ الحل الجذري (صف انتظار لكل رقم) = Phase C.

### 3.6 — نص الموديل يُرمى في فرع المنيو 🟡 → موثّق (سلوك مقصود)
عند [SEND_MENU] تُرسل الصور فقط؛ البرومبت يطلب "اكتب فقط [SEND_MENU]". لا تغيير.

# تعديلات المالك النهائية (2026-06-12) — مطبَّقة ومنشورة

أُضيفت لبرومبت v2 (المصدر [system-prompt-v2.txt](system-prompt-v2.txt)) وأُعيد
لصقها كاملة في عقدة AI Agent مع تحقّق diff تام (5345 محرف، مطابقة بايت-ببايت):

- **عنوان المطعم**: قسم جديد — "المطعم بدوار يحيى عياش، دخلة شركة أوريدو" (يُذكر
  لمن يسأل عن الموقع).
- **الدفع**: قسم جديد — كاش عند الاستلام أو التوصيل، ولا دفع إلكتروني. ولمن
  يسأل عن الفيزا/البطاقة: "للأسف لسا الدفع عالفيزا مش متوفر، بس كاش حالياً."
  (وحُذف ذكر "الموقع/طرق الدفع" من قسم "خارج الموضوع" لأنها صارت معروفة.)
- **أخذ رقم الزبون**: تدفق الطلب صار يطلب الرقم بكل الحالات — استلام: الاسم +
  الرقم؛ توصيل: الاسم + الرقم + العنوان. وأُضيف حقل `customer_phone` لـ JSON
  الـ [ORDER].

**تغيير مرافق في عقدة Parse Order** (مع تحقّق صحة نحوية):
الرقم اللي يعطيه الزبون يُلتقط من الـ JSON. عمود `customer_phone` بالـ DB يبقى
**رقم واتساب** (ضروري لبحث الحالة/الإلغاء وزر واتساب بالداشبورد). لو الرقم
المُعطى **مختلف** عن رقم واتساب (آخر 9 أرقام) → يُطوى في ملخّص الطلب كسطر
"رقم التواصل: ..." فيظهر للموظف بالبطاقة. لو نفس الرقم → ما في تكرار.
